# プロジェクト規約: アラート疲労解析

## 1. 基本原則 (Core Principles)
- **保存前の最終確認 (Surgical Time-out)**:
  - ファイルを新規作成、あるいは既存のファイルを書き換える際は、必ず「実施前に」変更内容の要約を提示し、ユーザーの許可を得ること。
  - 複雑なコードの場合は、まず「考え方（ロジック）」を提示し、合意を得てから執筆を開始すること。
  - 勝手にファイルを保存（Overwrite）して完了報告するのではなく、「この内容で保存して良いですか？」と必ず問いかけること。
- **確認の徹底**: 重要なファイルの削除、大規模なコード修正、またはディレクトリ構造の変更を行う前には、必ずユーザーに確認し許可を得ること。
- **データの保護**: `data/` フォルダ内のファイルは患者情報を含む可能性があるため、読み取りは最小限にし、外部への送信や不必要なコピーは厳禁とする。
- **破壊的変更の禁止**: 既存の解析結果やスクリプトを上書き・削除する際は、必ずバックアップ（Gitのコミット等）を提案すること。

## 2. 解析スタイル (Analysis Style)
- **Rコードの作法**: `tidyverse` スタイルを基本とし、パイプ演算子（`|>` または `%>%`）を活用した読みやすいコードを書くこと。
- **コメントの付与**: 統計手法（例：スプライン回帰、クラスタリング）を選択した理由は、医学的・統計的背景を含めてコード内にコメントすること。
- **再現性の担保**: 乱数（seed）の設定や、使用したパッケージのバージョン管理を意識すること。

## 3. ワークフロー (Workflow)
- **コミットの推奨**: 有意な進捗があった場合は、`claude commit` を提案すること。
- **エラーへの対応**: コード実行エラーが発生した場合は、単に修正するだけでなく、原因を特定し報告すること。
